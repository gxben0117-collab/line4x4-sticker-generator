# LINE 貼圖產生器 V4 重新設計提案（3 組方案）

版本：V1.00
產生日期：2026-07-22
狀態：**待選擇方向**——本檔只提案，不動 index.html。使用者選定方向（或混合）後才進入實作。

## 背景與限制條件（三方案共用）

使用者目標：產出的貼圖要能**真的拿去 LINE Creators Market 上架賣**，不是自己聊天用。核心限制：

1. **固定一次 16 張**（4×4），沿用現有生產模式。
2. **尺寸要符合 LINE 官方規定**（見下），現有 prompt 已經對，但目前只做了「16 張貼圖」，**沒有做主圖與標籤圖**——這兩張是上架必填，是目前產品的缺口，三個方案都要補。
3. 要有**多種字體**可選。
4. 要有**去背/不去背**兩種模式。
5. 預設背景用**青綠色**，方便事後去背。

### LINE 官方貼圖規格（提交 Creators Market 用）

| 項目 | 尺寸 | 格式 | 備註 |
| --- | --- | --- | --- |
| 貼圖本體 01～16.png | 最長邊不超過 370×320 px | PNG，需透明背景 | 每張 < 1MB；建議主體離邊界留白，避免縮圖時被裁到 |
| 主圖 main.png | 240×240 px | PNG | 縮圖代表圖，通常選最有代表性的一張角色姿勢 |
| 標籤圖 tab.png | 96×74 px | PNG，透明背景 | 貼圖分頁小圖示，畫面要極簡（縮到這麼小仍要看得懂） |

> 這組數字是 LINE Creators Market 行之有年的標準規格。正式送審前建議在 LINE Creators Studio 頁面上再核對一次，避免規則微調。

**目前產品缺口**：只產生 16 張貼圖的 prompt，main.png、tab.png 完全沒做。三個方案都會**新增「主圖 + 標籤圖」自動生成 prompt**，這是必補的，不管選哪個方案。

### 為什麼「青綠色底」是對的方向，而且要順便修一支腳本

專案裡已經有 `process-stickers` 技能會呼叫 `C:\AIProjects\003圖片暫存區\process.mjs` 做切割去背。我看了它現在的邏輯：它是抓「白色」當背景色，再用連通區塊分析去區分「外部白底」跟「角色內部的白色（例如眼白、白色衣服）」——這段邏輯特別複雜，就是因為**白底跟角色身上的白色會打架**，容易誤刪衣服或五官。

改用**高飽和度青綠色**當背景，這個顏色在人物插畫裡幾乎不會自然出現（不像白色），去背會準很多，連通區塊那套複雜判斷都可以簡化。所以這不只是「換個顏色」，我會建議：

- prompt 端：背景色改成固定青綠色，例如 `#00C896`（青綠、不偏純綠也不偏藍，AI 產圖時比較不會誤判成草地或水），色塊要求純色無漸層無陰影。
- `process.mjs` 端：`removeWhiteBg()` 改成可設定 key color 的版本，預設抓這個青綠色，白色偵測留著當備用選項（因為舊圖還是白底）。

這件事我會在使用者選定方案後一併做，不算額外任務。

### 有無背景怎麼設計

不是「開關」，是**三個背景模式**，因為用途不同：

| 模式 | 用途 | 適用方案 |
| --- | --- | --- |
| 青綠去背底（預設） | 標準貼圖，事後去背變透明 | A、C 全部；B 的角色主體 |
| 白底 | 舊流程相容，或使用者想自己在 Photoshop 摳 | 全部（保留選項） |
| AI 融合背景（不去背） | 背景本身就是畫面一部分（特效、場景） | 只有 B 會用到，而且**不是整格填滿的背景**，而是漂浮在角色周圍的特效元素（雷電、愛心、爆炸符號），外圍仍是透明——不然貼圖會變成一塊實心方塊，不能用 |

## 方案 A：人物主圖 + 常用文字（穩健路線，最接近現有產品）

**核心概念**：延續現有邏輯，一個角色 + 16 句短文字，但從「使用者亂選 16 句」升級成「一次做一整包主題貼圖」，例如「加班苦命包」「閨蜜互動包」「戀愛撒嬌包」——**16 句是同一個情境的完整敘事**，而不是東拼西湊。這樣做出來的貼圖包上架後比較容易被搜尋到、風格統一，買家一次買一個場景。

**16 張長什麼樣**（以「上班社畜包」為例）：早安上班、開會中、已讀、deadline到了、我想下班、崩潰中、辛苦了、下班啦！……角色姿勢、表情、字體風格全套統一。

**要改什麼**：
- 首頁改成先選「主題包」而不是先選單句（V3-lite 的分類系統可以直接變成主題包清單的基礎）。
- 每個主題包附「快速套用」，一鍵帶入 16 句 + 對應字體 + 對應背景色。
- 新增主圖／標籤圖 prompt。

**優點**：風險最低、工作量最小、現有 1404 句資料庫直接沿用、V3-lite 分類剛好可以變成主題包分類。
**風險**：視覺上跟市面上大量「角色+對話框文字」貼圖同質化高，賣相普通。
**適合**：想快、想穩、先測試市場水溫，或已經有固定角色 IP 想快速出多包。

---

## 方案 B：人物主圖 + 情緒特效（畫面感最強，最像「會賣」的貼圖）

**核心概念**：不是句子驅動，是**情緒驅動**。使用者選一個情緒（例如「震驚」），系統同時給出：表情/姿勢描述 + 對應背景特效（雷電、烏黑閃電紋、冷汗滴、爆炸符號、愛心特效），文字變成可選而非必須。這是目前 LINE 熱賣貼圖常見的做法——誇張表情＋視覺特效，縮圖階段就抓眼球，比純文字貼圖更有記憶點。

**16 張長什麼樣**（以「日常誇張情緒包」為例）：晴天霹靂（角色僵住+背景打雷）、暴怒（角色冒火+背景火焰紋）、狂喜（角色跳起+背景撒花）、社死（角色石化+背景裂痕）、暖哭了（角色落淚+背景光暈愛心）……每張是「表情+姿勢+特效」三件事綁在一起的一個情緒單位，而不是一句話。

**要改什麼**：
- 資料結構整個換掉：不用 `scriptData`（句子清單），改成「情緒卡」清單，每張卡是 `{情緒名稱, 角色表情動作, 背景特效描述, 建議文字(可留空)}`。這是全新的資料表，現有 1404 句可以部分回收當作「建議文字」欄位，但主結構要重建。
- UI 從「挑句子」變成「挑情緒卡」，卡片要顯示特效示意（用文字描述即可，不需真的畫圖）。
- prompt 組合邏輯要處理「特效元素透明底浮動」而不是整格背景填滿（見上方背景模式說明）。

**優點**：差異化最高，商業貼圖感最強，不受限於文字，甚至可以做「無字純情緒包」直接賣。
**風險**：工作量最大（新資料表要重新設計 20-30 張情緒卡，不是搬資料就好）；AI 生成角色+特效同框的一致性比純文字貼圖更難控制，可能需要多次調 prompt 才會穩定。
**適合**：想做真正差異化、有心力磨圖的人；不急著馬上量產很多包。

---

## 方案 C：文字當主體（量產最快，AI 失敗率最低）

**核心概念**：反過來，**文字是主角**，角色是配角或甚至不出現。例如「早安!!」用大字藝術字體+插畫感裝飾（放射線條、手繪箭頭、可愛塗鴉框），角色只是縮在角落比個手勢，或整張根本沒有角色。這類貼圖在 LINE 上很常見（純字貼圖包），做起來 AI 生成失敗率最低，因為**不需要維持角色臉部一致性**（這是目前 AI 生成貼圖最大的痛點），量產速度最快。

**16 張長什麼樣**：早安（大字+日出線條）、收到（大字+打勾圖示）、笑死（大字+爆笑星星）、辛苦了（大字+溫柔緞帶框）……16 張共用同一套字體與裝飾風格，字才是要辨識的主體，角色可有可無。

**要改什麼**：
- 字體選單變成**產品核心功能**，不是附屬設定：要準備多套完整風格（例如：手寫可愛體、粗體塗鴉、書法體、潮流貼紙體、簡約線條體），每套字體要配好對應的裝飾元素（線條/圖示/外框），不是單純換字型這麼簡單。
- 現有 1404 句資料庫直接拿來當「字面內容」候選，V3-lite 的分類（尤其熱門100）在這裡最好用——因為這方案要衝量，越常用越好賣。
- prompt 結構改成「文字+裝飾」為主、「角色」變成可選插槽。

**優點**：生產速度最快、失敗率最低、現有句子資料庫（尤其熱門100）直接就是最佳素材、最適合快速上架測水溫、鋪貨速度快。
**風險**：辨識度依賴字體與裝飾設計，如果字體風格做得普通，賣相跟一堆免費字體貼圖沒兩樣，要在裝飾細節上花心思才會好看。
**適合**：想最快量產多組包上架、先用低成本測哪些主題賣得動，之後再挑賣得好的主題做成方案 A 或 B 的加強版。

## 三方案比較

| | A 人物+文字 | B 人物+情緒特效 | C 文字主體 |
| --- | --- | --- | --- |
| 工作量 | 小（改現有資料結構） | 大（重建情緒卡資料表） | 中（字體+裝飾套組要準備） |
| AI 生成穩定度 | 中（角色一致性挑戰存在但已驗證可行） | 低（角色+特效同框更難控） | 高（無臉部一致性壓力） |
| 差異化/賣相 | 中 | 高 | 中，靠字體裝飾撐 |
| 現有 1404 句資料庫可用度 | 全部可用 | 部分可用（當輔助文字） | 全部可用，尤其熱門100 |
| 適合速度 | 中速 | 慢工出細活 | 最快 |

## 我的建議

如果要一次做到底、盡快有東西可以上架賣：**先做方案 C 打底**（量產快、失敗率低、現有句子庫直接派上用場），**同時把方案 A 需要的「主圖+標籤圖」「字體選單」「青綠背景+process.mjs 更新」這些共通基礎建設一起做**，因為三個方案都要用到。方案 B 因為資料結構要整個重建、AI 生成難度也高，適合等 C／A 先上架驗證過後，有心力再做深化款。

但這是我的判斷，不是定論——三個方向可以混，例如「16 張裡 12 張走方案 C 的量產款、留 4 張走方案 B 的重點特效款」也是可行的組合。你選一個方向、或告訴我怎麼混，我就照這個開始重做。
